The language that
lights the lantern.

wick is the lantern engine's own scripting language: Lua's size and feel, with the sharp edges designed out. Static types with optionals, flat records, type-checked engine calls, deterministic randomness, and a GC that only runs between frames — in ~2,100 dependency-free lines you can read in an afternoon.

Every Lua pain, fixed by design

wick exists because we shipped real games in Lua and catalogued what hurt. Each pain became a language decision — not a linter rule, not a convention, a thing the compiler enforces.

Optionals, enforced

lt.load_save() returns str? — using it unchecked is a compile error. Unwrap with ?? or narrow with if x != nil. The "empty save file crashes frame 1" bug we actually shipped in Lua cannot be written in wick.

Typos can't create globals

Every variable is a let. Assigning a misspelled name doesn't silently create a nil global — it doesn't compile.

Typed engine calls

Every lt.* signature is known to the compiler. Wrong arity, wrong type — you get the call name and argument number on the error screen at load, not a runtime surprise on frame 4,000.

No truthiness

Conditions must be bool. if 1 and if name don't compile; "" is not secretly true. 0-based indexing, continue, and str+str-only + round out the de-quirking.

Records, when earned

Flat record types with named fields — no methods, no inheritance. Admitted when KORA's kitchen props became five parallel lists. The blog is the paper trail.

GC at frame boundaries only

Collection runs exactly once per frame, after present — never mid-draw. And there's no JIT anywhere: iOS-legal by construction, with speed from static types (specialized opcodes, slot-resolved locals) instead.

Deterministic by default

rand() is a fixed xorshift — the same sequence on every machine. Combined with the engine's fixed-timestep mode, whole gameplay sessions replay byte-for-byte in CI.

The same game, twice

Kora Night ships in both languages as separate side-by-side projects that evolve together. The famous line — loading a hi-score that might be a 0-byte file:

-- Lua: shipped with a frame-1 crash.
-- "" is truthy, so `or "0"` never
-- fires; tonumber("") = nil poisons
-- everything downstream.
local best = tonumber(
  lt.load_save("best") or "0")
// wick: the only version that
// compiles. load_save is str?,
// num() is num? — both nil cases
// are handled, or it's a type error.
let best = num(
  lt.load_save("best") ?? "") ?? 0

The book

The wick Programming Language — a 65-page book, typeset in Latin Modern with LaTeX, is the recommended starting point: a real tutorial (you build a small game), the complete language reference, the optionals chapter with the full story of the bug that started it all, the engine interface, the internals of the compiler and VM, and an honest design-trade-offs chapter. Every code sample in it is valid v0.1 wick, verified against the reference implementation — the tutorial's final program compiles and renders unmodified. Download the PDF (free, zlib-licensed like everything else; LaTeX source in book/).

Part of the lantern engine

wick lives in the engine repo (wick/) and ships with it — there is nothing separate to install. A game is a folder with main.wick; hot-reload, the in-engine error screen, and CI screenshot verification all work exactly as they do for Lua and C, which remain first-class. Start at the documentation.
